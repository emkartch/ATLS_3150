Start with Accessibility and Usability

Think of accessibility and usability from the start, don't make accessibility part of a checklist.

Follow the 7 Principles of Universal Design

When designing something, physically or digitally, make the means of access the same (or VERY similar), consider variations in ability, make it simple and easy to understand, provide necessary information to the user (no matter ability level), minimize chances for hazard or error, make it comfortable to use, and provide space for anyone to be able to approach and use effectively.

How to Design an Accessible Website

Structure and Semantics

HTML Hierarchy

As mentioned on the "What" page of this website, HTML hierarchy is important to follow. Here is an example of how it might look if you follow it correctly: 

                                             
<h1>Heading 1</h1>

    <h2>Heading 2</h2>
    
        <h3>Heading 3</h3>

        <h3>Heading 3</h3>

    <h2>Heading 2</h2>

        <h3>Heading 3</h3>
    
    <h2>Heading 2</h2>

        <h3>Heading 3</h3>

Screen Reader Only

Also mentioned on the "What" page, you can use CSS to hide elements like headings so it doesn't take up space, but a screen reader can still detect and read it. Here is an example of what your code may look like:

HTML 

                                                 
<nav>

    <h2 class="sr-only">
        Main Navigation
    </h2>

</nav>

CSS 

                                           
.sr-only {
    border: 0;
    clip: rect(0, 0, 0, 0);
    height: 1px;
    margin: -1px;
    overflow: hidden;
    padding: 0;
    position: absolute;
    white-space: nowrap;
    width: 1px;
}

Bold vs. Strong Attributes

While they visually look the same, strong and bold both convey a different meaning to assistive technologies and code-reading systems, so it is important to use them correctly.

Bold
Use it for a piece of text where you want to draw attention without adding any extra importance or meaning.
Strong
Use it for a piece of text where you want to draw attention and add extra importance and meaning.

Using Native HTML elements

When creating a header, nav, main, aside, or footer for your website, use the native HTML elements instead of a span or a div. This allows for assistive technology and code-reading systems to more accurately understand your content.

Relationship between Native HTML elements and ARIA

It's the first rule of ARIA.

Accessible Forms

Ensure your website forms can be operated using keyboard alone.

Labels

A label describes the function of a form control (text area, checkbox, menu, etc.) and generally appears adjacent to it. The label element programmatically associates a text label to a form control, creating a connection in the code that doesn't rely on vision.

Intuitive Instructions

Make sure it is clear with use of color and shape when certain fields of the form are required to be filled out. For example, you may see that a lot of forms use “*” to indicate a required questions.

Include a grayed out example of the information needed in a field so users have a visual understanding of what you want. For example, a field that asks for your name and the greyed out example reads “Last Name, First Name” so users know to put their last name and a comma before their first name.

Accessible Tables

As your tables get a bit more complex in structure, it is useful to give them more structural definition. One clear way to do this is by using "thead," "tbody," and "tfoot," which allow you to mark up a header, body, and footer section for the table.

Note about Adding Structural Definition to Tables

These elements don't make the table any more accessible to screen reader users, and don't result in any visual enhancement on their own. Instead they help with any style specification you might want to do.

Make sure to add a caption to your table to describe the table's content and to use row and column headers.

Responsive and Accessible CSS Layout

CSS Grid and Flexbox

Use CSS grid and flexbox to create the layout for your page instead of workarounds because assistive technologies and code-reading systems will be able to more easily and effectively understand your content and its structure.

Mobile Accessibility

Design for mobile first, make your website accessible over a broad range of devices.

Account for things like touch screen use, small screen sizes, different input modalities, device use in different settings (like in the sunlight), and more.

Rich Internet Applications

ARIA

Follow the five rules of ARIA.

Adding an ARIA role overrides the native role semantics in the accessibility tree and indirectly affects what is reported to a screen reader or other assistive technology.

Adding an ARIA role will not make an element look or act differently for people not using assistive technology. It does not change the behaviors, states and properties of the host element but only the native role semantics.

Tools to Test for Accessibility

  • Nu HTML Checker
    • Checks your HTML for any coding errors that may result accessibility issues.
  • HTML5 Outliner
    • Shows a page outline of heading and sectioning elements so you can check your heirarchy and if there are any accidentally unnamed elements.
  • WAVE Evaluation Tool
    • WAVE is a web accessibility evaluation tool developed by WebAIM.org, it provides visual feedback about the accessibility of your web content.
  • Test your keyboard accessibility by attempting to navigate your website using only your keyboard.
  • Test your screen reader accessibility by going through your website with a screen reader.
